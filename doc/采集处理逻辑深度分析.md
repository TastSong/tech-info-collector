## 2. 第一层：站点采集方式选择

### 2.1 流程描述

```
输入: 站点URL列表 + render配置
  │
  ├─ render=static   → nativeFetch()     (原生 http/https 模块)
  ├─ render=dynamic  → fetchDynamic()    (Playwright Chromium)
  └─ render=lightpanda → fetchWithLightpanda() (Lightpanda CDP)
```

**对应文件**：`src/crawler/fetcher.ts`、`src/crawler/playwright.ts`、`src/crawler/lightpanda.ts`

#### static 模式（`nativeFetch`）
- 使用 Node.js 原生 `http`/`https` 模块，非 `undici fetch`
- 理由：支持 HTTP→HTTPS 跨协议重定向（如 moj.gov.cn）、可禁用 ECDHE 密码套件（兼容 stic.sz.gov.cn 等 SSL 异常站点）
- 指数退避重试（1s → 2s → 4s，最多 3 次）
- 自动编码检测（Content-Type header → `<meta charset>` → UTF-8 回退）
- SSL 回退策略：标准 TLS Agent → TLS_FALLBACK_AGENT（禁用 ECDHE/ECDH）
- 重定向 cookie 转发（防止 moj.gov.cn 等重定向循环）

#### dynamic 模式（`fetchDynamic`）
- Playwright 单例浏览器池：复用同一个 Chromium 实例
- 每次抓取独立 BrowserContext（隔离 cookie/缓存）
- 页面加载策略：`domcontentloaded` → 等待 `networkidle`（15s 超时，不致命）→ 额外 800ms 缓冲

#### lightpanda 模式（`fetchWithLightpanda`）
- 独立 Docker 服务，通过 CDP WebSocket 连接
- 与 Playwright 相同接口签名，使用 `chromium.connectOverCDP()`
- 连接超时 10s，避免 Lightpanda 不可用时永久挂起
- 注意：不支持 locale/geolocation/timezone 等 Emulation 域

#### 渲染模式自动检测（站点分析器专用，`site-analyzer.ts`）
- 策略：静态优先 → Lightpanda → Playwright 逐步降级
- 通过 `isMeaningfulHtml()` 判断：正文长度 > 200 字符 且 链接数 > 3，且无 JS 空壳特征
- 静态成功即返回，不继续尝试（最佳性能）

### 2.2 LLM 参与点

**本层无 LLM 参与**。渲染模式选择完全由确定性代码驱动：
- 运行时采集：依赖 `sites.render` 列的预配置值
- 站点分析器：依赖 `isMeaningfulHtml()` 的确定性规则（正文长度 + 链接数检测）

### 2.3 优点

1. **SSL 兼容性极强**：自定义 TLS Agent 禁用 ECDHE 密码套件、关闭证书验证，能抓取大多数国内政府/企业站点的不规范 HTTPS
2. **编码检测完善**：三级回退（header → meta → utf-8），覆盖国内常见的 GBK 编码站点
3. **渐进式渲染降级**（站点分析器）：静态→Lightpanda→Playwright，最大限度节省资源
4. **浏览器资源复用**：单例模式避免反复启动 Chromium，每个 context 隔离避免 cookie 串扰
5. **跨协议重定向支持**：原生 `http`/`https` 模块而非 fetch API，是适配国内站点生态的正确选择
6. **重试策略合理**：指数退避 + 仅对可重试错误（5xx/网络超时/连接拒绝）重试

### 2.4 缺点与改进建议

#### 问题 1：静态抓取缺少 JavaScript 渲染检测
当前 `nativeFetch` 对 SPA 站点只会拿到空壳 HTML。虽然 `sites.render` 可以配置为 `dynamic`，但如果是新增站点且未正确配置，会静默失败。

**建议**：在 runner.ts 的采集结果过少时（如候选链接 < 3），自动尝试 `lightpanda` 或 `dynamic` 回退：
```typescript
// 在 intelligentCrawl 中，prefilterLinks 结果 < MIN_CANDIDATES 时
// 自动尝试 dynamic 模式，而非仅尝试下一个 URL
if (candidates.length < MIN_CANDIDATES && input.render === "static") {
  // 自动升级为 lightpanda/dynamic 重试
}
```

#### 问题 2：Playwright 浏览器池缺少健康检查与自动恢复
`getBrowser()` 返回单例，如果浏览器崩溃（OOM/超时被 kill），后续所有 dynamic 任务都会失败，直到下次 `closeBrowser()` → 重新 `getBrowser()`。

**建议**：
```typescript
export async function getBrowser(): Promise<Browser> {
  if (browser && browser.isConnected()) return browser;  // 已有检查
  // 但 fetchDynamic 中 page.goto 超时不会触发重连
  // 增加：连续 N 次失败后自动 close + 重建
}
```

#### 问题 3：Lightpanda 缺少就绪检查
`getLightpandaBrowser()` 只在首次调用时连接，如果 Lightpanda 服务重启，已断开的 browser 对象仍被复用。

**建议**：在 `fetchWithLightpanda` 中捕获连接错误并自动重连：
```typescript
try {
  await page.goto(url, ...);
} catch (e) {
  if (isConnectionError(e)) {
    browser = null; // 强制下次重连
    // 重试一次
  }
}
```

#### 问题 4：编码检测不够全面
当前仅检测 `charset` 声明，但部分老旧站点使用非标准编码声明（如 `charset=gb2312` 但实际是 GBK）。

**建议**：对 `gb2312`、`gbk`、`gb18030` 统一使用 GBK 解码；增加 chardet 类启发式编码检测。

## 3. 第二层：站点信息采集（智能爬虫）

### 3.1 流程描述

本层是**LLM 参与的核心环节**，采用两阶段流水线：

```
Phase 1: 链接发现
  ├─ 多URL兜底 (最多3个) → fetchHtml → prefilterLinks (确定性Cheerio)
  ├─ LLM筛选: 候选链接JSON → generateText → 返回 {i: 索引}[]
  └─ 输出: 文章链接列表

Phase 2: 详情抓取
  ├─ queueFor (按域名限流) → Promise.allSettled 并行抓取
  ├─ parseDetail (确定性Cheerio) → 标题/正文/日期提取
  └─ 输出: 完整文章列表
```

**对应文件**：`src/ai/intelligent-crawl.ts`、`src/crawler/parser.ts`、`src/crawler/rate-limit.ts`、`src/crawler/intelligent.ts`

#### Phase 1 详细步骤

**1a. 多 URL 兜底策略**
```typescript
// 最多尝试 3 个 URL，选择"候选链接数最多"的那个
const urls = input.siteUrls.slice(0, 3);
for (let i = 0; i < urls.length; i++) {
  const candidates = prefilterLinks(await fetchHtml(url), url, scope);
  if (candidates.length > bestCandidates.length) {
    bestCandidates = candidates;
    bestUrl = url;
  }
  // 第一个 URL 已足够 (≥10条) → 立即停止
  if (i === 0 && candidates.length >= MIN_CANDIDATES) break;
}
```

**1b. 预筛选 (`prefilterLinks`)**
- 使用 Cheerio 解析 HTML，遍历所有 `<a href>` 标签
- **安全性过滤**：跳过 PDF/DOC/ZIP/图片等非 HTML 文件
- **噪声过滤**：
  - 前缀匹配：排除"关于/联系/登录/注册/版权/隐私/免责/首页/返回"等
  - 长度检测：≤6 字符的纯英文/数字 → 可能为导航/缩写
  - 广告过滤：排除"广告/特价/促销/秒杀/领取/优惠/红包/福利"等
- **去重**：同 URL 只保留一个
- **输出格式**：紧凑 `{i: 索引, t: 标题文本(100字), p: 父元素}` 列表

**1c. LLM 链接筛选**
```typescript
const llmResult = await generateText({
  model: getModel(),
  temperature: 0.1,
  system: "你是链接筛选器。从候选链接中选出属于文章/新闻/博客的链接...",
  prompt: `站点：${siteName}\n候选链接：${linksJson}\n返回 JSON 数组 [{i: 索引}]`,
  output: Output.text(),
});
```

LLM 角色：从预筛选的候选链接中进一步筛选出真正的文章链接。输入从原始 HTML (50-100KB) 压缩为紧凑 JSON (2-5KB)。

**1d. 结果解析 (`extractIndexArray`)**
- 四级回退解析：直接 JSON.parse → 代码块提取 → 正则匹配数组 → 空数组

#### Phase 2 详细步骤

**2a. 按域名限流**
```typescript
// 每域名独立 PQueue：concurrency=3, 每2秒最多3次请求
const tasks = articleUrls.map(item =>
  queueFor(item.url).add(async () => {
    const html = await fetchHtml(item.url, input.render, ...);
    const detail = parseDetail(html, { /* 空选择器 → 通用提取 */ });
    // ...
  })
);
```

**2b. 详情解析 (`parseDetail`)**
- **标题提取**：配置的 `titleSelector` → `h1` → `<title>` 三级回退
- **正文提取**：配置的 `bodySelector` → 通用主内容抽取（找含 ≥2 个 `<p>` 且文本最长的容器）→ body 纯文本
- **清洗**：移除 `script/style/nav/aside/form/.comment/.share/.breadcrumb`
- **截断**：正文最长 20000 字符
- **日期**：从配置的 `dateSelector` 提取

**2c. 通用主内容抽取 (`extractMainContent`)**
- 遍历所有 `div/article/section`，找含 ≥2 个 `<p>` 的容器
- 选择清洗后文本最长的
- 回退：整个 `body` 的纯文本

### 3.2 LLM 参与点分析

| 步骤 | LLM 角色 | 输入 | 输出 | 温度 | 超时 |
|------|---------|------|------|------|------|
| **1c. 链接筛选** | 从链接列表中识别文章链接 | 紧凑JSON (2-5KB, ≤100条) | `[{i: 索引}]` | 0.1 | 45s |
| **站点分析器 (site-analyzer.ts)** | DOM结构分析 + CSS选择器发现 | 清洗后HTML + 结构摘要 | 选择器JSON | 0.1 | 60s |

**LLM 的边界由确定性代码严格约束**：
1. **输入**：已预筛选的链接列表（非原始 HTML），最多 100 条
2. **输出**：仅索引数组，由 `extractIndexArray()` 确定性解析
3. **验证**：索引边界检查 `idx >= 0 && idx < candidates.length`
4. **回退**：LLM 失败 → 直接跳过该站点（不阻塞整个采集）

### 3.3 Prompt Injection 防护

`src/crawler/intelligent.ts` 中的 `sanitizeForLLM()` 过滤了以下注入模式：
```typescript
const INJECTION_PATTERNS = [
  /ignore (all )?(previous|above|prior|preceding) (instructions?|directives?)/gi,
  /(disregard|forget|override) (all )?(previous|above|prior) (instructions?|directives?)/gi,
  /you are (now|no longer) a[n]? /gi,
  /system (prompt|message|instruction|directive)/gi,
  /<\|im_start\|>/gi, /<\|im_end\|>/gi, /<\|system\|>/gi,
];
```

**注意**：此 sanitize 应用于链接文本（`c.text` 和 `c.parentClass`），但**未应用于 LLM 输出的 JSON 解析后的内容**——不过由于 LLM 只输出索引号（非自由文本），注入风险极低。

### 3.4 优点

1. **LLM 输入大幅压缩**：原始 HTML (50-100KB) → 紧凑 JSON (2-5KB)，token 消耗降低 95%+，延迟降低 80%+
2. **确定性 + LLM 协同设计优秀**：预筛选（规则）→ LLM（判断）→ 确定性解析（规则），三层互补
3. **LLM 输出安全**：仅返回索引数组，不信任 LLM 的任意文本输出
4. **多 URL 兜底策略务实**：取候选链接数最多的 URL，比"第一个失败就放弃"更鲁棒
5. **通用正文提取降级**：`extractMainContent()` 在无 bodySelector 时自动找最长文本容器，大幅降低逐站配置成本
6. **链接文本噪声过滤丰富**：覆盖中文导航/广告/促销等常见噪声模式
7. **按域名限流**：独立的 PQueue 防止打穿目标站点
8. **LLM 失败不阻塞**：Phase 1 LLM 失败仅跳过当前站点，Phase 2 并行抓取用 `Promise.allSettled` 容忍单条失败

### 3.5 缺点与改进建议

#### 问题 1：LLM 筛选的 scope（关注范围）实际未充分利用

```typescript
// intelligent-crawl.ts 第 246 行
`关注范围：${scope}`
```

`scope` 仅作为 system prompt 中的一句提示，没有结构化的匹配逻辑。LLM 可能忽略 scope 而凭"像文章"就选中链接。例如 scope 为"网络安全情报"的站点，LLM 仍可能选中该站的"公司团建新闻"链接。

**建议**：
```typescript
// 增强 system prompt 中的 scope 约束
`- 必须与关注范围密切相关，不相关的文章即使格式正确也应排除
 - 关注范围：${scope}
 - 示例：范围是"网络安全"时，排除"公司年会"、"员工福利"、"产品促销"等`
```

#### 问题 2：缺少链接有效性预检

`prefilterLinks` 仅检查 URL 扩展名和协议，不检查实际的 HTTP 可达性。Phase 2 中可能有大量 404/403，浪费抓取资源。

**建议**：在 Phase 1 LLM 筛选后、Phase 2 抓取前，增加 HTTP HEAD 请求的可选预检（可通过环境变量控制开关）：
```typescript
// 可选：对筛选后的 URL 做 HEAD 请求，排除明显的死链
// 但注意：部分站点对 HEAD 返回 405，需处理
```

#### 问题 3：通用正文提取可能命中非文章容器

`extractMainContent()` 的启发式方法（找含 ≥2 个 `<p>` 且文本最长的容器）在以下场景会误判：
- 页面侧边栏有长文本列表
- 评论区有大量文本
- 页脚有站点介绍

**建议**：
- 增加权重因子：位于 `<article>`、`<main>` 标签内的容器加权
- 排除已知噪声类名：`.sidebar`, `.comment`, `.footer`（已在清洗中移除标签，但 class-based 选择可能遗漏）
- 段落密度检测：计算 `<p>` 标签内文本占总文本的比例，正文区域通常在 70% 以上

#### 问题 4：LLM 筛选的 MAX_CANDIDATES 硬限制为 100

```typescript
const MAX_CANDIDATES = 100;
candidates.slice(0, MAX_CANDIDATES)
```

当页面有 200+ 候选链接时（如大型门户首页），硬截断可能丢失优质文章。

**建议**：
- 对链接按文本长度、关键词特征预排序（标题更长的优先？h 标签内的优先？）
- 或分页提交 LLM：第一批 100 条 → 如果结果 < MAX_ITEMS → 提交下一批

#### 问题 5：Phase 2 无重试机制

```typescript
const tasks = articleUrls.map((item) =>
  queueFor(item.url).add(async () => {
    // 单次 fetchHtml，失败直接返回 null
  })
);
```

虽然 `fetchHtml` 内部有重试（最多 3 次），但如果所有重试都失败（如临时 DNS 故障），该文章直接丢失。

**建议**：在 `intelligentCrawl` 层面增加一次整体重试（对失败的 URL 间隔 5s 后再试一轮），或至少将失败的 URL 记录下来供后续补偿采集。

#### 问题 6：`intelligent.ts` 中的 tool-calling 基础设施存在但未被使用

`src/crawler/intelligent.ts` 提供了完整的 tool-calling 基础设施：
- `SessionCache` — 会话级 LRU 缓存
- `createFetchPage` — fetch_page tool 工厂
- `extractLinksTool` — 按 LLM 指定的 CSS 选择器提取链接

但当前的 `intelligent-crawl.ts` **完全没有使用这些 tool**。实际的智能爬虫使用的是"预筛选 → LLM 一步筛选"的简化方案。tool-calling 方案虽然理论上更强大（LLM 可以多轮探索 DOM），但当前未被启用。

**建议**：
- 考虑对"难以一次性筛选"的复杂站点启用 tool-calling 模式
- 或者，如果确认简化方案足够，清理掉未使用的 tool-calling 代码以减少维护负担
- 当前两种方案并存可能导致未来的维护困惑

#### 问题 7：`scope` 参数未传入发布时间相关的过滤

LLM 筛选时不知道每篇文章的发布时间，可能在 Phase 1 就选入"上个月的旧闻"，然后在 Phase 3 AI 审核时才发现过时被 reject，浪费了 Phase 2 的详情抓取 tokens。

**建议**：如果站点有 dateSelector 且能在列表页提取日期，考虑将"发布日期"作为 prefilterLinks 的附加字段传给 LLM，让 LLM 在筛选阶段就能过滤明显过时的内容。

## 4. 第三层：采集后信息处理（AI 审核）

### 4.1 流程描述

```
输入: status=raw 的文章
  │
  ├─ analyzePending() 批量查询
  ├─ 按并发度 (默认3) 入队处理
  │
  ├─ 单篇处理:
  │   ├─ 状态标记: raw → analyzing
  │   ├─ reviewArticle()
  │   │   ├─ 输入截断: 标题≤200字, 正文≤6000字
  │   │   ├─ generateText (低温度 0.2)
  │   │   ├─ extractJson (容忍代码块)
  │   │   ├─ Zod schema 校验 (reviewSchema)
  │   │   └─ sanitizeDates (内容日期校验)
  │   ├─ decideStatus()
  │   │   └─ combinedScore = qualityScore*0.7 + newsScore*0.3
  │   │       < AI_PUBLISH_THRESHOLD → rejected
  │   │       !usable → rejected
  │   │       其余 → published
  │   ├─ contentDate 覆盖 publishedAt (LLM推断 > HTML解析)
  │   └─ 落库: aiReviews + 更新 articles.status
  │
  └─ 输出: published/rejected 计数
```

**对应文件**：`src/ai/analyze.ts`、`src/ai/sandbox.ts`、`src/ai/schemas.ts`

### 4.2 LLM 参与点分析

| 维度 | 详情 |
|------|------|
| **调用方式** | `generateText()`（JSON 文本模式），非 `generateObject()` |
| **模型** | OpenAI-compatible API，通过 `AI_BASE_URL`/`AI_API_KEY`/`AI_MODEL` 环境变量配置 |
| **温度** | 0.2（偏向确定性） |
| **输入截断** | 标题 200 字、正文 6000 字 |
| **输出 Schema (Zod)** | `relevant`, `summary`, `headline`, `keyPoints`, `tags`, `qualityScore`, `usable`, `isNews`, `newsScore`, `reason`, `contentDate` |
| **超时** | 无显式超时（依赖 SDK 默认值），analyze.ts 中无 timeout 包裹 |

### 4.3 沙盒安全设计

```typescript
// sandbox.ts 的沙盒边界
1. 输入硬截断（标题 200、正文 6000 字）    → LLM 看不到全文/全库/网络
2. 输出 Zod 强校验（reviewSchema）          → 格式不符即抛错
3. 低温度（0.2）                           → 降低随机性
4. 无工具集（v1）                          → LLM 无法调用任何外部工具
5. 确定性状态闸门（decideStatus）           → LLM 评分仅建议，最终决定由代码做出
6. 确定性日期校验（sanitizeDates）          → 防止 LLM 输出未来日期
```

**安全边界总结**：
```
┌──────────────────────────────────────┐
│          LLM 沙盒                     │
│  输入: 标题200字 + 正文6000字 + scope │
│  ┌──────────────────────────────┐    │
│  │  generateText (无工具)        │    │
│  │  → JSON 文本                 │    │
│  └──────────────────────────────┘    │
│         ↓                            │
│  确定性代码防线:                       │
│  ├─ extractJson: 容忍格式变异但必须合法JSON
│  ├─ Zod safeParse: 字段类型/范围强校验
│  ├─ sanitizeDates: 未来日期修正
│  └─ decideStatus: 综合评分闸门         │
└──────────────────────────────────────┘
```

### 4.4 状态闸门算法

```typescript
export function decideStatus(r: Review): "published" | "rejected" {
  if (!r.usable) return "rejected";                        // 硬闸门1: 不可用
  const t = threshold("AI_PUBLISH_THRESHOLD", 0.5);        // 默认 0.5
  const combined = r.qualityScore * 0.7 + r.newsScore * 0.3; // 综合评分
  if (combined < t) return "rejected";                     // 硬闸门2: 质量不足
  return "published";
}
```

**权重设计意图**：
- `qualityScore × 0.7`：情报价值占主导
- `newsScore × 0.3`：新闻属性为辅助（教程/分析/评论也可发布）
- `usable=false` → **无条件拒绝**（噪声/空壳/纯导航/公告）

### 4.5 优点

1. **安全问题考虑周全**：
   - 输入硬截断防止 context 溢出和 prompt 注入放大
   - 输出 Zod 校验防止格式逃逸
   - 日期校验防止"未来日期"污染
   - 最终状态由确定性代码决定，LLM 仅提供建议

2. **结构化输出设计良好**：
   - `qualityScore` 和 `newsScore` 正交评分，细分情报价值与新闻属性
   - `contentDate` 从正文推断实际事件日期（而非网站发布日），语义上更准确
   - `headline` 用于展示，`summary` 用于预览，`keyPoints` 用于快速浏览，各司其职

3. **审计留痕**：`aiReviews` 表完整记录每次 LLM 调用的输入/输出/评分/tokens，便于回溯和成本核算

4. **状态机设计合理**：`raw → analyzing → published/rejected`，`analyzing` 中间态防止并发重复处理

5. **失败处理稳健**：单篇审核失败 → 直接标记 `rejected`，不阻塞批次中其他文章

6. **内容日期去幻觉**：`sanitizeDates()` 检测未来日期并修正，防止 LLM 幻觉导致的时间线污染

### 4.6 缺点与改进建议

#### 问题 1：使用 `generateText` + 手动 JSON 解析而非 `generateObject`

当前方案：
```typescript
const { text } = await generateText({ ..., output: Output.text() });
const parsed = reviewSchema.safeParse(extractJson(text));
```

虽然注释说明"兼容网关不支持 response_format / tool 结构化输出"，但 `extractJson` 的正则解析在以下场景可能失败：
- LLM 在 JSON 前后输出大段解释文本
- JSON 中包含嵌套对象/数组较大导致正则不完整匹配
- 特殊字符转义问题

**建议**：
- 检测网关是否支持 `response_format: { type: "json_object" }`，支持时优先使用
- 或者使用 Vercel AI SDK 的 `generateObject()` + Zod schema（SDK 内部处理结构化输出）
- 对 `extractJson` 失败的情况，增加更详细的错误日志（当前仅抛出"未找到 JSON 对象"）

#### 问题 2：没有显式的审核超时控制

```typescript
// analyze.ts 中
const r = await reviewArticle({...});  // 无超时包裹
```

如果 LLM API 响应缓慢（或网关卡住），单篇文章可能永久阻塞队列中的一个槽位。虽然 PQueue 不设整体超时，但长时间等待会阻塞整个批次。

**建议**：
```typescript
const r = await withTimeout(
  reviewArticle({...}),
  60_000  // 单篇审核最多 60 秒
);
```

#### 问题 3：缺少去重审核（相同/高度相似文章重复调用 LLM）

目前每篇 `status=raw` 的文章都会触发一次 LLM 审核。如果两篇文章内容高度相似（如转载、多站同步发布），会产生重复的 LLM 调用和成本。

**建议**：
- 在审核前，通过 `contentHash` 检查是否已有相同内容的审核结果
- 如果已有相同 hash 且已审核，直接复用评分和摘要
```typescript
// 在 reviewArticle 前检查
const existingReview = db.select().from(schema.aiReviews)
  .innerJoin(schema.articles, ...)
  .where(eq(schema.articles.contentHash, article.contentHash))
  .limit(1).all().at(0);
if (existingReview) { /* 复用 */ }
```

#### 问题 4：`contentDate` 推断的准确性无法保证

LLM 被要求从正文推断"事件实际发生日期"，例如"1月5日消息" → `2026-01-05`。但这里有几个问题：
- "上周五"需要 LLM 知道"今天是什么日期"来做推算，系统 prompt 中提供了 `todayDateStr`，但 LLM 不一定正确使用
- 跨年推断：如果当前是 2026-01-03，正文提到"12月28日"，LLM 可能误判为 2026-12-28（未来）而非 2025-12-28

`sanitizeDates()` 会将未来日期修正为今天，但这只是兜底修复而非正确推断。

**建议**：
- 在 system prompt 中增加跨年推断的明确规则："若推断日期看似晚于今天，应考虑可能是上一年"
- 或者在 `sanitizeDates` 附近增加智能修正：如果推断日期在"未来但不超过 3 个月前"（新文章提前发布的可能），可能是当年；如果"距今超过 7 个月后"，大概率是上一年

#### 问题 5：综合评分权重不可配置

```typescript
const combined = r.qualityScore * 0.7 + r.newsScore * 0.3;
```

0.7/0.3 的权重硬编码。某些场景（如纯新闻聚合）可能希望提高 `newsScore` 权重；某些场景（如技术分析聚合）可能希望提高 `qualityScore` 权重。

**建议**：通过环境变量配置：
```typescript
const qw = Number(process.env.AI_QUALITY_WEIGHT ?? 0.7);
const nw = Number(process.env.AI_NEWS_WEIGHT ?? 0.3);
const combined = r.qualityScore * qw + r.newsScore * nw;
```

#### 问题 6：缺乏人工干预入口

当前审核只有自动的 `published`/`rejected` 两种结果。实际运营中可能需要：
- "待人工审核"状态（qualityScore 在临界值附近）
- "置顶"或"精选"标记
- "标记为误判"的反馈机制

**建议**：
- 增加一个"灰度区"：`combinedScore` 在 `AI_PUBLISH_THRESHOLD` 到 `AI_PUBLISH_THRESHOLD + 0.15` 之间的文章标为 `review_needed` 而非直接拒绝
- 增加 `articles.reviewedBy` 字段，支持人工审核覆盖

## 5. 第四层：调度与编排

### 5.1 流程描述

**对应文件**：`src/pipeline/service.ts`、`src/pipeline/runner.ts`、`src/scheduler/cron.ts`、`src/pipeline/abort.ts`

#### 服务层编排 (`service.ts`)
```
runCrawl()
  ├─ 1. 查询目标站点 (single siteId OR all enabled)
  ├─ 2. 过滤 aiInvolvement != 'none'
  ├─ 3. 创建 crawl_session
  ├─ 4. 按域名分组 (groupByHost)
  │     └─ 同域名 → 串行 (避免反爬)
  │     └─ 跨域名 → 并行 (PQueue, 默认 concurrency=10)
  ├─ 5. 等待所有站点完成
  ├─ 6. 关闭浏览器资源
  ├─ 7. 汇总 session 结果
  └─ 8. autoAnalyze? → analyzePending()
```

#### 单站点采集 (`runner.ts`)
```
runSite(site)
  ├─ 1. 创建 run_log 记录 (status=running)
  ├─ 2. 检查 aiInvolvement != 'none'
  ├─ 3. 调用 intelligentCrawl()
  ├─ 4. queryExisting(urls) — 按 URL 精确查询已入库
  ├─ 5. deduplicateAndSave() — contentHash 对比
  │     ├─ 同站同hash → 跳过
  │     ├─ 同站不同hash → 更新
  │     ├─ 跨站同URL → 跳过(重复)
  │     └─ 新URL → 插入
  └─ 6. finalizeRun() — 更新 run_log 和 sites.lastRunAt
```

#### 定时调度 (`cron.ts`)
```
startScheduler()
  ├─ 从 settings.cron_interval 读 cron 表达式
  ├─ node-cron 按 cron 定时触发
  ├─ 每分钟检测 cron 变更并热更新
  └─ maybeCatchUp() — 启动时检查是否错过触发窗口(5分钟内补执行)
```

#### 中止机制 (`abort.ts`)
```
createAbortController() — 新一轮采集前创建
getAbortSignal()       — 各层透传检查
abortCrawl()           — 中止 + 清空 PQueue
```

### 5.2 优点

1. **分层清晰**：service → runner → intelligentCrawl，每层职责单一
2. **域名分组并行策略合理**：同域名串行防反爬，跨域名并行提效率
3. **中止机制完整**：AbortController 透传到底层 fetch，PQueue 清空
4. **去重策略分层**：URL 主键（数据库层）+ contentHash（应用层），双保险
5. **Cron 热更新**：每分钟检测配置变更，无需重启
6. **补执行窗口**：5 分钟内补执行，覆盖"cron 触发时容器未启动"的场景

### 5.3 缺点与改进建议

#### 问题 1：补执行窗口 (5分钟) 可能不足

如果容器在 cron 触发 6 分钟后才启动（如大镜像下载 + 启动），本次定时采集将被跳过。

**建议**：增加"今天是否已执行"的持久化记录，而非仅依赖 5 分钟窗口：
```typescript
// 检查今天是否已有成功的 crawl_session
const todayStart = new Date(); todayStart.setHours(0,0,0,0);
const todaySession = db.select().from(schema.crawlSessions)
  .where(and(
    eq(schema.crawlSessions.status, 'success'),
    gte(schema.crawlSessions.endedAt, todayStart)
  )).limit(1).all().at(0);
if (!todaySession) { /* 补执行 */ }
```

#### 问题 2：大站采集无超时控制

单个站点的 `runSite` 无整体超时。如果某个站点响应极慢（如 LLM 超时 45s × 多次重试 + 大量详情页抓取），可能阻塞整个同域名组。

**建议**：在 `runSite` 外层增加站点级总超时（如 5 分钟），超时后记录日志并继续下一个站点。

#### 问题 3：采集结果缺少内容质量指标

当前 `runSite` 返回 `{fetched, skipped, updated, errorCount, status}`，但没有"平均正文长度"、"有效文章率"等质量指标，不利于监控采集质量下降。

## 6. 综合评估与改进路线图

### 6.1 LLM 参与全景

| 阶段 | LLM 参与 | 调用频率 | Token 消耗 | 失败影响 |
|------|---------|---------|-----------|---------|
| 站点分析 (site-analyzer) | ✅ 选择器发现 | 仅新增站点时 | 高 (~5-15K tokens) | 回退确定性方法 |
| 链接筛选 (intelligentCrawl Phase 1) | ✅ 文章链接识别 | 每站点每次采集 | 中 (~2-5K tokens) | 跳过该站点 |
| 详情提取 (Phase 2) | ❌ 确定性 Cheerio | — | 0 | — |

| 状态判定 (decideStatus) | ❌ 确定性代码 | — | 0 | — |

**总计**：每篇入库文章约消耗 **1-3K tokens**（审核阶段）+ 站点级平摊的链接筛选 tokens。

### 6.2 架构优点总结

3. **渐进式降级**：LLM 失败 → 确定性回退 → 默认兜底，不会全盘崩溃
4. **代码质量良好**：模块化清晰、注释详尽、重试/限流/错误处理周全
5. **国内站点适配深入**：SSL 兼容、GBK 编码、跨协议重定向，体现了对实际场景的理解
