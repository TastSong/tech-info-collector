## 100+ 网站信息采集工具 — 架构设计与主要流程

------

### 一、整体架构（分层设计）

Code

```
┌─────────────────────────────────────────────────────────┐
│                    控制层 (Control Layer)                  │
│  ┌─────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐  │
│  │ 调度器   │  │ 配置管理  │  │ 监控告警  │  │ API/CLI │  │
│  └─────────┘  └──────────┘  └──────────┘  └─────────┘  │
└─────────────────────────────────────────────────────────┘
                          │
┌─────────────────────────────────────────────────────────┐
│                   采集层 (Crawl Layer)                     │
│  ┌─────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐  │
│  │ 任务队列 │  │ Worker池  │  │ 代理池   │  │ 限流器  │  │
│  └─────────┘  └──────────┘  └──────────┘  └─────────┘  │
└─────────────────────────────────────────────────────────┘
                          │
┌─────────────────────────────────────────────────────────┐
│                  解析层 (Parse Layer)                      │
│  ┌─────────────┐  ┌───────────┐  ┌───────────────────┐  │
│  │ 规则引擎     │  │ 插件系统   │  │ AI 智能提取(可选) │  │
│  └─────────────┘  └───────────┘  └───────────────────┘  │
└─────────────────────────────────────────────────────────┘
                          │
┌─────────────────────────────────────────────────────────┐
│                 存储层 (Storage Layer)                     │
│  ┌──────┐  ┌────────┐  ┌──────────┐  ┌───────────────┐  │
│  │ DB   │  │ 文件存储 │  │ 缓存(Redis)│  │ 消息队列     │  │
│  └──────┘  └────────┘  └──────────┘  └───────────────┘  │
└─────────────────────────────────────────────────────────┘
```

------

### 二、核心模块设计

#### 1. 站点配置模块

- 每个网站一个配置文件（YAML/JSON），定义：
  - 目标 URL 模式
  - 采集频率与时间窗口
  - 请求方式（静态 HTTP / 动态渲染）
  - 解析规则（CSS 选择器 / XPath / JSON Path）
  - 认证方式（Cookie / Token / 登录流程）
  - 限流策略（QPS、间隔）

#### 2. 调度器 (Scheduler)

- 基于优先级的定时任务调度
- 支持 Cron 表达式
- 失败重试与退避策略（指数退避）
- 依赖关系管理（某些页面需要先登录）

#### 3. 采集引擎 (Crawler Engine)

- **静态采集**：HTTP Client（aiohttp / httpx）处理普通页面
- **动态渲染**：Headless Browser（Playwright / Puppeteer）处理 JS 渲染页面
- **代理池**：轮换 IP 避免封禁
- **并发控制**：每站独立限流，全局并发上限

#### 4. 解析引擎 (Parser Engine)

- **插件化设计**：每个站点一个解析插件
- **规则引擎**：通用规则（CSS/XPath）+ 自定义脚本
- **数据清洗**：去重、格式化、校验
- **可选 AI 提取**：对非结构化内容用 LLM 提取关键信息

#### 5. 存储模块

- **结构化数据** → PostgreSQL / MySQL
- **原始页面快照** → 对象存储（S3 / MinIO）
- **缓存层** → Redis（去重指纹、频率限制）
- **消息队列** → RabbitMQ / Kafka（解耦采集与处理）

#### 6. 监控与告警

- 采集成功率、延迟、数据量统计
- 站点变更检测（页面结构变化自动告警）
- 封禁检测与自动切换策略

------

### 三、主要流程

Code

```
1. 启动 → 加载 100+ 站点配置
         ↓
2. 调度器按频率/优先级将任务推入队列
         ↓
3. Worker 从队列获取任务
         ↓
4. 判断采集方式：
   ├── 静态页面 → HTTP Client 请求
   └── 动态页面 → Headless Browser 渲染
         ↓
5. 反爬处理：
   ├── 选择代理 IP
   ├── 设置 Headers / Cookies
   └── 处理验证码（人工/打码服务）
         ↓
6. 获取原始响应 → 存储快照（可选）
         ↓
7. 解析引擎根据站点规则提取数据
         ↓
8. 数据清洗 → 去重 → 格式校验
         ↓
9. 写入数据库 + 发送变更通知
         ↓
10. 更新监控指标 → 失败则重试/告警
```

------

### 四、技术选型建议

| 层级      | 推荐方案                                            |
| --------- | --------------------------------------------------- |
| 语言      | Python（生态丰富）或 Go（高并发）                   |
| HTTP 采集 | httpx / aiohttp（Python）；colly（Go）              |
| 动态渲染  | Playwright（推荐）/ Puppeteer                       |
| 任务队列  | Celery + Redis（Python）；自建 goroutine pool（Go） |
| 调度      | APScheduler / 自定义 Cron                           |
| 数据库    | PostgreSQL（结构化）+ Redis（缓存/去重）            |
| 消息队列  | RabbitMQ（中小规模）/ Kafka（大规模）               |
| 监控      | Prometheus + Grafana                                |
| 容器化    | Docker + Docker Compose（部署）                     |

------

### 五、100+ 站点的关键设计考量

1. **插件化** — 新增站点只需添加配置+解析规则，无需改核心代码
2. **隔离性** — 单个站点故障不影响其他站点采集
3. **可观测性** — 每个站点独立统计成功率/延迟/数据量
4. **弹性扩展** — Worker 数量可水平扩展
5. **变更适应** — 站点结构变化时自动检测并告警，降级为 AI 提取
6. **合规性** — 遵守 robots.txt，合理设置采集频率，避免对目标站点造成压力
